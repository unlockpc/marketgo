# MarketGo 使用手册（桌面版）

> MarketGo（Go-To-Market Automation，原名 UnMarket）：全自动 / 半自动 全球营销获客工具 —— 基于真实浏览器（Unzoo）、用你自己的账号、读真帖再回复、可人工审核、7×24 运行。
> 本手册对应**桌面应用**（Tauri，`src-tauri` 里的 Rust 引擎 + 前端界面）当前已实现的功能，**没实现的会明确标注**，方便你逐条核对"是不是这么回事"。
> 纯命令行用法另见 [README.zh.md](./README.zh.md)。

---

## 一、它到底是干嘛的（一条获客漏斗）

不是几个孤立功能，而是一条**从养号到转化**的流水线，每一步都在防封号：

```
养号(地基) → 健康体检 → 发帖前防封闸门 → 读真帖 → 买家意向打分+闸门
        → 生成"像真人"的回复 → 半自动审核/全自动发布 → 线索 → 转化 → 效果分析
```

| 阶段 | 干什么 | 自动化 |
|------|--------|--------|
| 养号 | 模拟真人浏览，把账号养出权重，避免新号一上来就被封 | 全自动（引擎按策略调度） |
| 健康体检 | 查账号是否还登录着 / 是否被封 | 全自动 |
| 防封闸门 | 养号期未满 / 健康异常 / 版块禁推广 → 不发 | 全自动 |
| 关键词回复 | 按关键词搜帖 → 读真帖 → AI 判定意向+生成回复 → 带货获客 | 半自动审核 或 全自动 |
| 评论回复 | 回复**自己帖子**下别人的评论（社区运营，低风险） | 半自动 或 全自动 |
| 发布 | 自己发帖/发文 | 全自动 |
| 线索/转化 | 每条发出的回复=一条线索，跟踪谁回应/转化 | 记录自动，标记手动 |
| 效果分析 | 漏斗 + 各平台/关键词 ROI | 自动统计 |

## 二、第一次使用前的准备

1. **Unzoo 浏览器在运行**：本工具所有浏览器动作都走 Unzoo（REST API 在 `http://127.0.0.1:9399`）。先确保 Unzoo 已启动。
2. **以 Gmail 为单位建"身份"并登录**：当前账号体系**以邮箱（身份/persona）为单位** —— 一个 Gmail = 一套独立浏览器(profile)+IP+指纹，名下各平台账号全共用它。去左侧「📧 邮箱账号」页（旧版的"身份隔离"页已并入这里）：点「+ 新建 Gmail」建身份 → 在该邮箱面板点「📧 登录Gmail」打开它的浏览器登好 Gmail → 「🚀 检查并开通账号」逐平台自动登录/注册。工具不替你保管密码——它用的是已登录浏览器配置(profile)的登录态。
3. **配一个 AI Key**（生成回复、意向打分要用）：设置页填 Gemini / OpenAI / DeepSeek / Qwen 任一的 Key。当前**默认 provider 是 Gemini**（默认模型 `gemini-2.0-flash`），每个 provider 的 Key 单独存。
4. **建好产品**：产品页填产品名 + 一句话标语（回复里软提产品时用）。

> 没登录平台、没配 AI Key 也能跑，但：未登录 → 养号/回复会被判"未登录"拦下；没 AI Key → 回复退化成模板（不推荐）。

## 三、怎么启动

**推荐（带界面）：** 在项目根目录执行
```bash
npm run tauri:dev
```
`tauri:dev` 会自动起前端静态服务器（端口 1420，即 `npx serve dist/tauri -l 1420 -s`）再开窗口。

**如果你看到窗口里写"localhost 拒绝连接 / ERR_CONNECTION_REFUSED"**：说明是用 `cargo run` 起的、但 1420 服务器没起。要么改用 `npm run tauri:dev`，要么单独起一个：
```bash
npx serve dist/tauri -l 1420 -s
```
然后点窗口里的「刷新」。（引擎是 Rust，独立运行；白屏只是网页没找到，不影响后台。）

**让引擎开机自动跑**：设置环境变量 `UNMARKET_ENGINE_AUTOSTART=1` 再启动；否则在任务页点「▶ 启动引擎」。

## 四、核心概念（搞懂这几个就会用了）

- **身份(persona) + Profile 继承**（统一在「📧 邮箱账号」页操作，邮箱视角）：
  - **身份 = 一个 Gmail**：在「📧 邮箱账号」页点「+ 新建 Gmail」以邮箱为单位建身份，每个身份独享一套浏览器 profile + IP + 指纹；归到这个身份名下的账号，**自动共用**它的 profile。多号隔离就靠它。
  - **邮箱视角**：页面顶部是「📧 邮箱」切换条，选哪个邮箱、下面就只显示那个邮箱名下的账号；未挂到任何邮箱的账号归到「🧩 未归属」tab。
  - **归属与解绑**：在账号卡片上给账号选一个身份即归属；选「-- 使用全局 Profile --」即解除归属、回落到全局默认。
  - **全局默认 profile**：在**设置页**选一个 profile 点「连接」会成为全局默认，供没有任何归属的账号兜底。
  - profile 解析优先级：**账号所属身份的 profile > 账号自身旧式绑定 > 该平台任一已绑定账号 > 全局默认**。四者都没有才报"无可用 profile"。
- **养号期 (warmup)**：账号注册后的"保护期"（默认 14 天，Twitter/Reddit/LinkedIn 均为 14）。**养号期内禁止对外带货回复 / 发帖**，只允许养号 + 回自己帖评论。这是防封的关键，可用 `engine_warmup_gate` 总开关控制。
- **健康度 health_status**：`healthy`（健康）/ `logged_out`（掉登录）/ `shadowbanned`（限流）/ `banned`（封禁）/ `unknown`（待配置）。掉登录/限流/封禁的账号会被自动跳过。
- **买家意向分 intent (0–100)**：AI 判断帖主"是否在主动找/评估解决方案"。低于阈值（默认 40）不回复。
- **回复模式 reply_mode**：
  - `review` 半自动（**默认**）：AI 生成回复后进**审核队列**，你批准了才发。
  - `auto` 全自动：直接发布（无人值守）。
- **演练 dry_run**：开了之后所有动作只"走流程+打日志"，**不真实发布**。第一次跑强烈建议先开演练看效果。
- **两种回复分型**：
  - 关键词回复（`reply_keyword`）：去**陌生人**帖子下回复获客，受养号期/意向/防封闸门约束。
  - 评论回复（`reply_mention`）：回复**自己帖子**下的评论，低风险、不受养号期约束。

## 五、完整使用流程（按这个顺序走）

### 步骤 1：配置（设置页 + 产品页 + 身份/账号页）
1. 设置页：填 AI Key（选 provider，默认 Gemini）。
2. 产品页：新建产品（名字 + 标语）。
3. 「📧 邮箱账号」页：「+ 新建 Gmail」建身份 → 「📧 登录Gmail」→ 「🚀 检查并开通账号」逐平台登录/注册；或在单个账号卡片点「🔑 自动登录」（查登录→Google 登录→否则注册）。确认账号已归到对应邮箱、对应平台已登录。

### 步骤 2：先养号（自动，地基）
- 启动引擎后，**养号调度会自动工作**：按平台策略 + 号龄分期（新号轻 / 成长重 / 成熟维持），在活跃时段自动安排账号浏览。
- 去**任务页 → 🌱 养号调度**面板看：每个账号的「分期 / 号龄 / 今日 x/目标 / 累计时长 / 健康徽标」。
- 你也可以手动催一把：账号里点养号，或批量「新建养号任务」。

### 步骤 3：配置获客活动（推广活动页）
- 新建 campaign，关联产品 + 平台 + 关键词。启动后会生成回复任务进任务队列。

### 步骤 4：启动引擎，先演练
- 任务页点「▶ 启动引擎」。
- **建议先开 dry_run 演练**（设置/配置里），看日志里生成的回复内容质量，确认像不像人、产品提得软不软。

### 步骤 5：半自动审核（默认、推荐）
- 关掉演练、保持 `review` 模式。引擎会：搜帖 → 读真帖 → 意向打分 → 生成回复 → **进审核队列**（不发）。
- 任务页 →「💬 回复模式」下方**审核队列**：每条带**意向分**（按意向高低排序）、原帖链接、AI 理由。
- 你可以**直接改文案**，然后「批准发布」或「驳回」。

### 步骤 6：转全自动（可选，确认稳了再开）
- 「回复模式」下拉切到「全自动」。引擎会直接发布（仍受限流/养号期/健康/版规闸门保护）。

### 步骤 7：看线索与效果
- **🎯 线索/转化**：每条发出的回复都成一条线索，标记「转化 / 已回应 / 忽略」。
- **📊 效果分析**：发现/跳过/回复/待审/线索/**转化**漏斗，按平台 + 热门关键词看哪儿 ROI 高。

## 六、任务页各面板速查（核对用）

| 面板 | 作用 |
|------|------|
| 引擎徽标 + ▶启动/⏸停止 | 引擎开关与状态（已处理数） |
| 💬 回复模式 下拉 | 半自动审核 / 全自动 切换 |
| 审核队列（待审 N） | 待批准的回复，带意向分，可编辑/批准/驳回 |
| 🌱 养号调度 | 每账号 分期/号龄/今日进度/累计/健康 |
| 📊 效果分析 | 漏斗卡片 + 各平台表 + 热门关键词 |
| 🎯 线索/转化 | 线索看板，标记转化/已回应/忽略 |
| 🔍 选择器自检 | 填平台+关键词，实时报命中数+样本，校准帖子选择器 |
| 任务列表 | 全部任务及状态，可解除阻塞/重试/取消 |

任务状态：⏳pending 等待 / ⚡running 执行中 / ✅completed 完成 / ❌failed 失败 / ⚠️blocked 阻塞(看原因) / 🚫cancelled 取消。

## 七、防封机制（为什么有时它"不发"）

工具默认**保守**，宁可不发也不冒险被封。被拦时任务会变 `blocked` 并写明原因：

1. **养号期未满**："养号期未满（x/y 天）" → 等号养熟，或在养号策略调小该平台 `warmup_days`（也可用 `engine_warmup_gate` 总开关临时关闭闸门）。
2. **健康异常**："账号健康异常(logged_out / shadowbanned / banned)" → 去该账号所属身份的浏览器重新登录。
3. **版块禁推广**（Reddit）：发评论前查 `/r/<版块>/about/rules.json`，禁自我推广/广告的版块直接跳过。
4. **低意向**：意向分 < `engine_intent_min`（默认 40）→ 跳过，省额度。
5. **安静时段 / 回复间隔**：可设安静时段（`engine_quiet_start/end`，夜间不对外发，已生效）；回复间最小间隔也会校验。每平台每日上限主要用于**养号调度**配额，并非对外回复的硬闸门。

> 这也是相对纯 API 类工具的优势：真实浏览器 + 养号 + 闸门，更难被风控。

## 八、配置项速查（存在 config 表 / nurture_strategies 表）

| 配置 | 含义 | 默认 |
|------|------|------|
| `engine_dry_run` | 演练模式（1 开 / 0 关） | 未设=关 |
| `engine_reply_mode` | review 半自动 / auto 全自动 | review |
| `engine_intent_min` | 意向阈值（低于不回） | 40 |
| `engine_warmup_gate` | 养号期闸门总开关（1 开 / 0 关） | 1 |
| `engine_quiet_start` / `engine_quiet_end` | 安静时段（整点，仅抑制对外发） | 未设 |
| 环境变量 `UNMARKET_ENGINE_AUTOSTART` | =1 开机自动跑引擎 | 未设 |

**养号策略 `nurture_strategies`（表默认 warmup 14 / 场次 2–5 / 单次 60–300 秒 / 活跃 9–22；各平台初始化为）：**

| 平台 | warmup_days | 每日场次 | 单次秒数 | 活跃时段 |
|------|-------------|----------|----------|----------|
| Twitter/X | 14 | 2–4 | 60–180 | 9–22 |
| Reddit | 14 | 2–5 | 60–300 | 9–23 |
| LinkedIn | 14 | 1–3 | 60–120 | 8–20 |

> 数据库：跨平台存在系统应用数据目录下的 `unmarket/unmarket.db`（用 Rust `dirs::data_dir()` 解析）。
> Windows：`C:\Users\<你>\AppData\Roaming\unmarket\unmarket.db`；macOS：`~/Library/Application Support/unmarket/unmarket.db`；Linux：`~/.local/share/unmarket/unmarket.db`。

## 九、常见问题 FAQ

- **Q：界面白屏 / localhost 拒绝连接？** A：用 `npm run tauri:dev`，或单独 `npx serve dist/tauri -l 1420 -s` 后点刷新（见第三节）。
- **Q：回复任务全被 blocked"养号期未满"？** A：符合预期——新号先养。等号龄到 warmup_days，或临时把该平台 warmup_days 调小测试。
- **Q：提示"未登录 xx 平台"？** A：去该账号所属身份(Gmail)的浏览器里手动登录该平台，再让健康体检/养号重跑。
- **Q：选择器自检命中 0？** A：多半是没登录（登录墙）或该平台改版。先登录再检测；改版就更新该平台的帖子选择器。
- **Q：回复内容质量怎么看？** A：开 dry_run 演练，日志会打印 AI 真实生成的回复 + 意向分 + 理由；或直接看审核队列。
- **Q：怎么彻底不自动发？** A：保持 `review` 模式（默认）——所有对外回复都要你点「批准发布」。

## 十、当前能力与边界（实话实说）

**已实现并验证：**
- 身份(persona) 以 Gmail 为单位：建身份 → 登 Gmail → 逐平台"检查并开通"（有就登录、没有就注册），名下账号共用其浏览器/IP/指纹。
- 养号调度引擎（号龄分期 + 活跃时段 + 间隔均摊）、账号健康体检（带轮询，修了 SPA 误判）。
- 发帖前防封闸门（养号期 / 健康 / Reddit 版规）。
- 关键词回复：搜帖 → 读真帖 → 意向打分 → 生成"针对本帖、像真人、按需软提产品"的回复（实测对 Reddit/Twitter 生成质量良好，甚至能按帖子语言作答）。
- 半自动审核队列（带意向分、可编辑、批准/驳回）、全自动发布。
- 线索/转化轻 CRM、效果分析漏斗。
- 选择器自检；Reddit/Twitter 选择器已校准；LinkedIn 走"信息流就地评论"（TipTap 编辑器，输入后才出现发布按钮）。

**边界 / 待接续：**
- 平台覆盖逐步增加；选择器需登录态、平台改版需重新校准。
- `reply_mention`（回自己帖评论）目前**仅 Reddit** 支持抓评论，其余平台优雅阻塞。
- LinkedIn 浏览器自动化风险较高（平台 2025 封锁过此类），建议低频 + 充分养号。
- 线索的"对方是否回应"目前**手动标记**；自动回访检测待做。
- 私信/外联序列、代理 IP 池（`proxies` 表已存在、可增删测，但**尚未接入**发布引擎）尚未联动。
- 全自动模式真实发布前，建议先用**你自己可控的帖子**点一次「批准发布」验证发布动作 OK 再放量。
